### Зберігання паролів в БД
В задачі вказано, що користувачу потрібно повертати pass, тому зберігати пароль у вигляді 
хешу недоцільно, оскільки з хешу пароль назад не отримати. Зберігати пароль у БД у 
відкритому вигляді можна для тестування, оскільки це зручно. Але для реальних умов 
зберігати пароль в БД у відкритому вигляді небезпечно. У випадку, коли пароль необхідно 
повертати користувачу, пароль варто шифрувати секретним ключем. <br>

Оскільки в завданні не вказані подробиці, в якому вигляді потрібно зберігати пароль і 
що саме повертати користувачу, то було розроблено ***3 різні стратегії*** щодо 
зберіганння пароля: 
<br/>


1) **Plain password strategy** - зберігати пароль у відкритому вигляді. Дану стратегію 
не варто використовувати на працюючому проекті, її можна використовувати для тестування. 
Це може бути зручно, оскільки можна відразу додати пароль у БД без модифікацій. 
Крім того, видно що саме потрапляє в БД при роботі. 
<br>


2) **Encrypted password strategy** - зберігати пароль в БД в зашифрованому вигляді. 
Пароль шифрується і розшифровується за допомого секретного ключа. У випадку, якщо 
зловмисник отримає доступ лише до даних в БД, то в нього не буде секретного ключа для 
розшифровки даних паролю.
<br>


3) **Hashed password strategy** - на мою думку найбільш безпечний підхід і рекомендований 
підхід для збереження паролю. За можливості краще використовувати даний варіант і не повертати 
користувачу поле `pass` в API. Для сумісності з іншими стратегіями при виборі даного методу 
користувачу повертається пустий рядок. Бо повертати хеш пароля замість пароля не бачу 
сенсу.

### Як вибрати/змінити стратегію зберігання пароля з 3-х доступних 
По замовченню увімкнута стратегія **Encrypted password strategy**, тобто пароль буде 
зберігати в БД у шифрованому вигляді.  
Для того, щоб вибрати одну з трьох стратегій, необхідно у файлі 
```shell
application/config/services.yaml
```
знайти інтерфейс 
```shell
App\Application\Password\PasswordStrategyInterface
```
 і призначити йому один з трьох сервісів, наприклад:
1) щоб вибрати **Plain password strategy**, необхідно зазначити
```shell
App\Application\Password\PasswordStrategyInterface: '@App\Application\Password\PlainPasswordStrategy' 
```
2) щоб вибрати **Hashed password strategy**, необхідно зазначити
```shell
App\Application\Password\PasswordStrategyInterface: '@App\Application\Password\HashedPasswordStrategy' 
```
3) щоб вибрати **Encrypted password strategy**, необхідно зазначити
```shell
App\Application\Password\PasswordStrategyInterface: '@App\Application\Password\EncryptedPasswordStrategy' 
```

### Як оновити тестові дані, щоб пароль був збережений у відповідному форматі  
Оновити тестові дані можна або за допомогою команди або вручну.  
Оновлювати тестові дані за допомогою команди необхідно після того, як було змінено 
стратегію для збереження пароля. Послідовність дій:
1) очистити таблицю від попередніх тестових даних
```shell
docker compose exec -it application php bin/console db:clear-users-and-tokens-tables
```
2) додати тестові дані
```shell
docker compose exec -it application php bin/console db:create-test-data
```
Тестові дані також можна оновити в ручному форматі:
1) почисти таблицю `users` і `tokens`
2) виконати sql з відповідного файлу в папці `application/dump` (по назвах файлів можна зрозуміти, який файл вибрати)
